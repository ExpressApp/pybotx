# FAQ и Best Practices

## Архитектура и зависимости

**Вопрос: Как не допустить зависимостей снизу вверх (infrastructure → application)?**

- Держите shared state и модели в domain.
- DTO для внешних API храните в infrastructure contracts.
- В presentation используйте inbound contracts.

**Вопрос: Где хранить бизнес-логику?**

- В application (use-cases) и domain (модели и инварианты).
- В presentation только парсинг/валидация и вызов сервисов.

## DI и конфигурация

**Вопрос: Что должно быть в контейнере?**

- Всё, что относится к wiring: порты, адаптеры, клиенты, сервисы.
- Минимизируйте логику в контейнере.

**Вопрос: Как безопасно менять зависимости?**

- Используйте `providers.Override` или `providers.Object`.
- Сохраняйте интерфейсы через порты.

## HTTP и ретраи

**Вопрос: Когда включать retry?**

- Для временных ошибок сети и 5xx.
- Не ретрайте операции без идемпотентности.

**Рекомендации:**

- Ограничивайте `max_attempts`.
- Используйте jitter.
- Избегайте ретраев для больших стримов, если нет защиты от повтора.

## Сообщения и builders

**Вопрос: Почему лучше использовать MessageOptions?**

- Единообразие API.
- Меньше параметров в вызовах.
- Упрощает bulk операции.

**Вопрос: Когда нужен builder, а когда send_message?**

- Builder нужен, когда сообщение собирается из нескольких шагов или есть общий options.
- `send_message` удобно, когда всё известно заранее.

## Bulk операции

**Вопрос: Как выбрать max_concurrency?**

- Начинайте с 2–5.
- Увеличивайте, если API стабилен и нет rate limits.
- Снижайте при ошибках 429 или таймаутах.

## Виджеты

**Вопрос: Когда использовать WidgetSession?**

- Когда элементы большие и metadata тяжёлая.
- Когда нужен diff без хранения списка в payload.

**Вопрос: JSON serializer или pickle?**

- JSON безопаснее, ограничен по типам.
- Pickle удобен, но менее безопасен для внешних данных.

## Безопасность

**Вопрос: Можно ли включать demo-команды в проде?**

- Нет. Используйте `TODO_DEMO_ENABLED=false`.
- Для тестов включайте `TODO_DEMO_ALLOW_RISKY=true` только в dev.

## Тестирование

**Рекомендации:**

- Unit-тесты для domain и application.
- Интеграционные тесты для adapters.
- Контроль слоёв через pytestarch.
